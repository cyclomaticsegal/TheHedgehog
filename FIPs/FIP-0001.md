# FIP-0001: The Hedgehog

| Field       | Value                                      |
|-------------|--------------------------------------------|
| FIP         | 0001                                       |
| Title       | Rename app to The Hedgehog and introduce tabbed workspace |
| Status      | Partially Superseded by FIP-0002           |
| Created     | 2026-04-03                                 |
| License     | MIT                                        |

## Abstract

Rename the application from "Regime Shift Dashboard" to **"The Hedgehog"**. Restructure the UI from a single-view dashboard into a tabbed workspace with three tabs:

1. **Regime Shift** — the existing dashboard (VIX monitoring, commodity correlation, AI analysis)
2. **Dexter** — embedded terminal running the [Dexter](https://github.com/virattt/dexter) autonomous financial research agent, with structured data integration
3. **51Folds** — integration with [51Folds.ai](https://51folds.ai) neuro-symbolic models

## Motivation

The dashboard currently runs AI analysis via Claude/GPT against market data already loaded into the app. As we integrate additional analytical tools — Dexter for autonomous financial research and 51Folds for neuro-symbolic modelling — the single-view layout no longer suffices. A tabbed workspace organises these capabilities into focused views while keeping them under one roof. The rename to "The Hedgehog" reflects this broader scope beyond regime shift detection alone.

## UI Structure

The top-level UI will present three tabs:

| Tab           | Description |
|---------------|-------------|
| **Regime Shift** | The entire current dashboard — VIX charts, commodity correlation, spike detection, AI analysis panel, activity log. No functional changes; it moves wholesale into this tab. |
| **Dexter**       | Embedded terminal (via `egui_term`) running the Dexter agent, plus a panel displaying parsed results from Dexter's `.dexter/scratchpad/` JSON files. |
| **51Folds**      | Integration with 51Folds.ai neuro-symbolic models for structured, probabilistic financial analysis. |

## Dexter Integration — Options Considered

### Option 1: Spawn and capture stdout

Use `std::process::Command` or `portable-pty` to run Dexter (`bun start`) as a child process. Pipe stdin/stdout back into the app to send queries and parse responses programmatically. Display the interaction in an egui panel.

**Pros:** Simple to implement, no changes to Dexter required.
**Cons:** Output is conversational/unstructured, harder to parse reliably.

### Option 2: Read Dexter's scratchpad

Spawn Dexter in the background and watch its `.dexter/scratchpad/` directory for JSON output files using a file watcher (e.g. the `notify` crate). Parse the structured JSON into the app's data model as Dexter works and display results alongside VIX/commodity charts.

**Pros:** Structured data, no Dexter modifications needed, non-intrusive.
**Cons:** One-directional (read-only), no interactive control over Dexter from the UI.

### Option 3: Embedded terminal with data integration

Embed a terminal panel in the dashboard (via `egui_term`) so the user can interact with Dexter visually. Simultaneously watch the `.dexter/scratchpad/` JSON files to pull structured results into the dashboard's existing panels (e.g. feed Dexter's financial analysis into the AI analysis panel and saved inferences).

**Pros:** Full interactive control, structured data integration, best of both worlds.
**Cons:** More complex implementation, adds `egui_term` dependency.

## Decision

**Option 3: Embedded terminal with data integration.**

This provides interactive access to Dexter within the dashboard while also capturing structured results for integration with existing analysis panels. The terminal panel gives the user full control over Dexter queries, and the scratchpad watcher feeds results into the dashboard's data model automatically.

## Dexter Integration — Detailed Implementation Plan

### Prerequisites

- **egui_term 0.1.0** — published on crates.io, depends on `egui ^0.31.0`. Compatible with our eframe 0.31. No egui upgrade required.
- **notify 9.0** — cross-platform filesystem watcher crate for monitoring the scratchpad directory.
- **Dexter** must be installed separately (`bun install` in the Dexter repo). The Hedgehog does not bundle Dexter.

### New Dependencies

```toml
egui_term = "0.1"
notify = "9.0"
```

### Dexter Scratchpad Format

Dexter writes JSONL files to `.dexter/scratchpad/YYYY-MM-DD-HHMMSS_<hash>.jsonl`. Each line is a JSON object with a `type` field:

| Type          | Description |
|---------------|-------------|
| `init`        | Records the original query |
| `tool_result` | Tool invocation: tool name, arguments, raw result, LLM summary |
| `thinking`    | Agent reasoning step |

### Architecture

```
┌─────────────────────────────────────────────────────────┐
│  Dexter Tab                                             │
│  ┌──────────────────────────┬──────────────────────────┐│
│  │                          │                          ││
│  │   Terminal Panel         │   Scratchpad Panel       ││
│  │   (egui_term)            │   (parsed JSONL viewer)  ││
│  │                          │                          ││
│  │   Runs: bun start        │   Watches:               ││
│  │   in Dexter directory    │   .dexter/scratchpad/    ││
│  │                          │                          ││
│  │   User interacts with    │   Displays structured    ││
│  │   Dexter directly        │   tool_result entries    ││
│  │                          │   as cards               ││
│  └──────────────────────────┴──────────────────────────┘│
└─────────────────────────────────────────────────────────┘
```

### Step 1: Add dependencies to Cargo.toml

Add `egui_term = "0.1"` and `notify = "9.0"` to `[dependencies]`.

### Step 2: Data model for scratchpad entries

Add to `src/models.rs`:

```rust
#[derive(Debug, Clone, Deserialize)]
#[serde(tag = "type")]
pub enum DexterEntry {
    #[serde(rename = "init")]
    Init { timestamp: String, query: String },
    #[serde(rename = "tool_result")]
    ToolResult {
        timestamp: String,
        #[serde(rename = "toolName")]
        tool_name: String,
        summary: Option<String>,
    },
    #[serde(rename = "thinking")]
    Thinking { timestamp: String, content: String },
}
```

### Step 3: Scratchpad file watcher

Create a background thread using `notify::RecommendedWatcher` to watch the `.dexter/scratchpad/` directory. On file create/modify events, read the JSONL file line by line, parse each line as a `DexterEntry`, and send results to the main thread via `mpsc::channel`.

New fields on `DashboardApp`:

```rust
// Dexter tab state
dexter_backend: Option<egui_term::TerminalBackend>,
dexter_pty_rx: Option<Receiver<egui_term::PtyEvent>>,
dexter_path: String,              // path to Dexter repo
dexter_scratchpad: Vec<DexterEntry>,
dexter_watcher_rx: Option<Receiver<Vec<DexterEntry>>>,
```

### Step 4: Embed the terminal via egui_term

In `show_dexter_tab()`, replace the placeholder with:

1. **Terminal initialization** — on first visit or button click, spawn a `TerminalBackend` with shell command `bun start` and working directory set to the Dexter repo path. Use `BackendSettings { shell: "bun", args: vec!["start"], .. }`.
2. **Terminal rendering** — render `TerminalView` in the left/central panel, passing the backend. Poll `dexter_pty_rx` for `PtyEvent::Exit` to detect when Dexter finishes.
3. **Scratchpad panel** — right side panel showing parsed `DexterEntry` items as cards: init queries as headers, tool results with tool name + summary, thinking steps in muted text.

### Step 5: Dexter path configuration

Add a `dexter_path: String` field to `AppSettings` (persisted to SQLite). In the Dexter tab, show a path input field and "Start Dexter" button when no terminal is active. The file watcher watches `{dexter_path}/.dexter/scratchpad/`.

### Step 6: Bridge scratchpad results to Regime Shift tab

Optionally allow tool_result entries (e.g. financial data lookups) to appear in the Regime Shift AI panel or saved inferences. This is a future enhancement and not required for initial integration.

### Sequence

1. Add crate dependencies (Cargo.toml)
2. Add data model types (models.rs)
3. Add scratchpad watcher module (new file: src/dexter.rs)
4. Add Dexter state fields to DashboardApp (app.rs)
5. Replace Dexter tab placeholder with terminal + scratchpad UI (app.rs)
6. Add Dexter path to AppSettings and persist it (models.rs, storage.rs)
7. Wire up polling in update() — add `self.poll_dexter()` alongside existing poll calls

### Risks

| Risk | Mitigation |
|------|------------|
| `egui_term` 0.1.0 is very new (Apr 2025), may have bugs | Pin exact version; terminal is non-critical, app works without it |
| Dexter not installed on user's machine | Graceful fallback: show instructions to install, disable Start button |
| Scratchpad directory doesn't exist yet | Create it on first watch, or only start watching after Dexter spawns |
| JSONL parsing fails on unexpected fields | Use `#[serde(deny_unknown_fields)]` or permissive deserialization with `serde_json::Value` fallback |
